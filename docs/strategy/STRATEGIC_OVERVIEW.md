# Strategic Overview

## Diagnosis

The repo already has substantial pricing code, tests, artifacts, and WRDS pipeline infrastructure. The current bottleneck is not feature count. The bottleneck is evidence freshness and public claim alignment.

The project should now operate in portfolio/research benchmark mode: use the existing library and pipeline to produce current, public-safe, impressive evidence.

## Current Strategic Direction

Run an Evidence-Backed Prestige Phase:

1. Current-HEAD evidence recovery and claim audit.
2. WRDS/OptionMetrics panel protocol and aggregate results.
3. Reproducible performance benchmark and optimization frontier.
4. Portfolio/release narrative only after evidence is current.

## What Matters Most

- Evidence-backed prestige.
- WRDS data leverage.
- Performance benchmarks with hardware/protocol context.
- Claim/evidence discipline.
- Avoiding process/documentation loops.

## Major Tradeoffs

- Move fast, but do not fabricate validation.
- Optimize for performance/prestige, but do not overstate unsupported results.
- Use WRDS aggressively, but keep raw licensed data out of git/logs/bundles.
- Treat Heston honestly if it is near parity with BS or if QE remains biased.

## Avoid

- Broad stale-doc cleanup.
- README polish before evidence refresh.
- Live WRDS overclaiming.
- Scenario/date/tolerance tuning after seeing results.
- Architecture rewrites without benchmark/research payoff.
- Packaging/release work before claim audit.

## Target State After Next Phase

The repo should have current-HEAD metrics, a clear claim/evidence map, safe WRDS sample/live boundaries, a larger WRDS or data-readiness package, and a benchmark/performance frontier ready for public presentation.
